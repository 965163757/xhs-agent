# Huashu Design Brief · 小红书创作助手原型
> 日期：2026-05-15  
> 依据：当前 `frontend/src` 代码、已有产品主色与真实路由结构  
> 交付：高保真 HTML 原型，不连接后端

## 设计假设
1. 这是一个面向持续生产的 **Agent 工作台**，不是单点写作工具。
2. 用户最关心三件事：AI 在做什么、笔记在哪里、图片/诊断任务是否还能恢复。
3. 管理员视角是真功能，不是边缘功能，因此用户归属、跨用户笔记/对话需要在信息架构里显性存在。
4. 图片素材在原型中只做诚实占位，不伪造真实用户图；生产界面中应显示真实 `/static/images/...` 缩略图。

## 位置四问
- **叙事角色**：产品信息架构和核心工作流总览。
- **观众距离**：桌面浏览器 60–90cm；需要高信息密度但不能牺牲可读性。
- **视觉温度**：温暖、轻、可信；保留小红书红但避免营销化。
- **容量估算**：每个主页面必须至少表达 3 个真实产品差异化点：Agent 工具链、笔记上下文、后台任务或管理员能力。

## 三套方向
### A · Josef Müller-Brockmann 式信息建筑（采用）
- 网格清晰、信息密度高、状态可读。
- 最适合承载 Agent trace、模型队列、后台任务和编辑工作台。

### B · Kenya Hara 式纸感留白（吸收）
- 让编辑区像一张可写的纸，降低普通用户心理负担。
- 适合内容编辑、手机预览、图片队列局部。

### C · Charles Eames 式创作控制室（吸收）
- 把任务、评分、图像、诊断作为创作系统的一部分展示。
- 适合任务中心、诊断页、管理员未来团队版扩展。

## 最终取舍
采用 A 为骨架，吸收 B 的温暖纸感背景和 C 的任务可观察性。原因：当前系统最需要解决的不是「看起来像社媒 App」，而是「AI 操作复杂时仍然可控」。
